# OpenPatch v0.1.0-alpha Checklist

This checklist is for the first public alpha release of OpenPatch.

The goal of `v0.1.0-alpha` is not to ship the full product vision. The goal is to ship one honest, working flow that open-source users can try end to end:

- onboard a machine with the CLI
- start the local worker
- verify worker and model connectivity
- open a private repository locally
- ask a read-only repository question
- receive a response through the web UI

## Release Scope

The alpha release should present OpenPatch as:

- a hosted UI plus local worker architecture
- a product with a real read-only repository flow
- an open-source early release for technical users

The alpha release should not present OpenPatch as:

- a finished editing agent
- a polished desktop installer
- a production-ready team deployment platform

## Functional Checklist

- `npm install -g openpatch` works for the published CLI package
- `openpatch onboard` completes and writes config under `~/.openpatch`
- `openpatch worker start` launches the local worker successfully
- `openpatch doctor` reports worker and model status clearly
- `openpatch status` reports current worker URL, provider, and repo base directory clearly
- `curl http://127.0.0.1:8000/health` returns a healthy response
- `POST /repo/open` works for a private GitLab repository with stored provider config
- `POST /agent/run` returns a structured read-only response
- the web UI can open a repository through the local worker
- the web UI can submit a read-only question and render the response

## Product Messaging Checklist

- README presents the read-only flow first
- README includes a Quickstart section
- README includes current capabilities and current limitations
- onboarding docs match the current CLI and worker behavior
- demo docs match the current read-only flow
- editing features are described as incomplete or in progress
- repository-source support is described consistently as GitLab, GitHub, and local projects for repository open flows
- `project_path` is used consistently as the repository identifier

## Documentation Checklist

- README links to onboarding and demo docs
- README links to the alpha checklist
- troubleshooting guidance covers worker startup failures
- troubleshooting guidance covers port conflicts
- troubleshooting guidance covers missing model and provider-permission problems
- CLI and web docs describe the same read-only alpha flow

## Verification Checklist

- verify a local Ollama setup with `qwen2.5-coder:7b`
- verify a private GitLab repository open flow
- verify the local worker health endpoint manually
- verify the web UI read-only flow manually in a browser
- verify CLI commands fail fast with clear errors when the worker is unavailable

## Post-Alpha Follow-Up

After `v0.1.0-alpha`, the next product steps should focus on:

- safer editing review flows
- validation and test execution UX
- explicit commit and push workflows
- clearer hosted-to-local worker pairing for non-local deployments
